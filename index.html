02 My role

My Contribution

I owned the security and access-control thinking for the team and turned it into a written HTML security guide. My work answered one question for every feature: who is allowed to do this, and what stops everyone else?

Access model

Defined the roles and their boundaries

Mapped what a customer, a producer, and an administrator are each permitted to do, so permissions are explicit rather than assumed.

Authentication

Designed sign-in on Supabase Auth

Used Supabase's authentication instead of hand-rolled login, so identity and sessions rest on a proven, ready-made system.

Authorization

Specified Row-Level Security rules

Each database row is scoped to its owner — a producer can manage only their own order items, and private records stay private even if the interface is bypassed.

Threat thinking

Modeled how the system could be misused

Looked past intended use to impersonation and unauthorized access, then tied each misuse to a control that blocks it.

Documentation

Wrote the security & access-control guide

Produced the team's HTML reference explaining the access rules and the reasoning behind them, so the design can be reviewed and maintained.

03 Foundations

What I Learned

Before designing access control, I worked through a module on engineering for security — the difference between safety and security, how breaches actually happen, and how to argue that a system is defensible. These are the ideas the rest of this page is built on.

01

Safety vs. security

Safety assumes accidents in a benign environment. Security assumes an intelligent, adaptive adversary in a hostile one who will probe your defenses until they break.

02

The CIA triad

Confidentiality, integrity, and availability are the load-bearing dimensions — preventing unauthorized disclosure, modification, and withholding of service.

03

The vocabulary of risk

Asset, threat, vulnerability, attack, control, exposure — a shared language that turns vague worry into something you can analyze and mitigate.

04

Anatomy of an attack

A threat meets a vulnerability, a control either holds or fails, and the attack either stops or proceeds to real exposure. Weak authentication is where it begins.

05

Use vs. misuse cases

You cannot only design for what users should do. You have to explicitly model the malicious interactions an attacker would attempt.

06

Defense in depth

Never rely on a single point of protection. Layer controls so an attacker must defeat platform, application, and record-level barriers in turn.

07

The control funnel

Avoid the vulnerability first, detect and neutralize what gets through, then limit exposure and recover — because the cost of failure rises at each stage.

08

'Shall not' requirements

Functional requirements say what a system shall do; security says what it shall not do. Proving that negative is the core challenge.

09

The assurance case

Security can't be proven absolutely, so you build a documented body of evidence — design rationale, threat analysis, and testing — arguing the system can withstand expected attacks.

04 The design

Access Control Design

Usaruna's access control is layered, so no single mechanism is a single point of failure. An attacker would have to get past identity, then role, then the record's own ownership rule.

Platform · Authentication
Application · Roles
Record · RLS
  • 1

    Platform level — authentication

    Supabase Auth verifies identity and manages the session. No valid identity, no entry.

  • 2

    Application level — authorization

    A verified user's role (customer, producer, or admin) decides which areas and actions are available.

  • 3

    Record level — Row-Level Security

    Even with the right role, RLS confines each user to their own rows — a producer touches only their own order items.

A single security mechanism is a single point of failure. Each layer assumes the one before it could be bypassed.

Each principle from the foundations maps onto a concrete decision in the platform. The left side is the theory; the right side is what we actually did.

06 The rulebook

Design Guidelines I Followed

From the ten secure-design guidelines, these are the ones that shaped Usaruna's access control most directly.

G1

Base decisions on explicit policy

Access rules come from a defined model of roles and permissions, not ad-hoc patches added when something breaks.

G2

Don't rely on one point of protection

Authentication, authorization, and RLS reinforce each other rather than standing alone.

G3

Default to locked

If a permission check can't pass, access is refused — the safe state is closed.

G4

Balance security with usability

Controls are strict but invisible in normal use, so users aren't tempted to work around them.

G5

Apply need-to-know

Compartmentalize data so a breach in one account can't reach everyone else's records.

G6

Plan for recovery

Assume a breach can happen and design so the system can be restored and trust re-established.

07 What it taught me

Security isn't a feature you add — it's an argument you have to keep proving.

Working on access control taught me that the hard part isn't writing a login screen. It's thinking like the attacker: assuming the environment is hostile and that someone will try to reach data they shouldn't.

I learned to put the rule where it can't be bypassed — in the database with Row-Level Security — instead of trusting the interface to behave. And I learned why a single control is never enough: real protection comes from layers that each assume the one before them might fail.

Most of all, I understood why you can never call a system "completely secure." The best you can do is build a clear, documented case — roles, misuse cases, and controls — that shows the system can withstand the attacks you expect. That mindset is what I carried into Usaruna.
